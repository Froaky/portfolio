# Portfolio Roadmap - Mateo Nahuel Coca

## Purpose

This document defines a practical, high-signal portfolio plan tailored to Mateo Nahuel Coca's current profile:

- Backend and fullstack developer
- ERP / Odoo customization experience
- API integration experience
- SQL-backed systems experience
- Internal tools and business process automation background
- Mobile app integration exposure via Flutter
- Remote-first job search focused on backend, fullstack, systems, and implementation-oriented roles

This is not a generic "personal website" plan.

The objective is to build a portfolio that:

1. makes Mateo look credible for backend/fullstack remote roles
2. proves real-world delivery, not tutorial work
3. shows product thinking and systems thinking
4. creates reusable assets for CV, interviews, LinkedIn, and job applications
5. can be built incrementally without waiting for a perfect design or massive scope

---

## Strategic Positioning

### Core positioning statement

Mateo should be positioned as:

**"A practical backend/fullstack engineer who connects business operations with reliable software: APIs, ERP customizations, internal tools, SQL-backed systems, and process automation."**

That framing is stronger than trying to compete as:

- a pure FAANG-style algorithm engineer
- a design-heavy frontend specialist
- a deep infra/platform specialist
- a pure AI researcher

### What the portfolio should sell

The portfolio should repeatedly communicate these messages:

1. Mateo ships software that solves real operational problems.
2. Mateo can design and implement systems across backend, frontend, data, and integrations.
3. Mateo understands business workflows, not just isolated code tasks.
4. Mateo is useful in remote environments because he can own work end to end.
5. Mateo is not a tutorial-driven junior; he has practical delivery experience.

### What the portfolio should avoid

The portfolio should avoid over-indexing on:

- flashy animation-heavy frontend with no substance
- fake startup copy with no real proof
- generic CRUD apps with no operational context
- inflated "AI engineer" branding without real systems depth
- vague claims like "passionate developer" with no concrete evidence

## Recommendation: Best Stack for This Portfolio

### Recommended stack

**Next.js + TypeScript + Tailwind CSS + MDX + simple i18n routing**

### Why this is the best fit

This stack is the strongest choice if the portfolio needs to do two jobs at once:

1. look modern and polished
2. act as proof that Mateo can build serious fullstack/frontend-capable software

### Why not plain HTML/CSS/JS

It can look good, but it signals less technical leverage and is slower to scale once case studies, bilingual content, and reusable sections grow.

### Why not Astro as first choice

Astro is excellent for content-heavy sites, but Next.js does a better job if the portfolio itself should help sell modern frontend/fullstack capability in the current market.

### Suggested stack details

- **Framework:** Next.js App Router
- **Language:** TypeScript
- **Styling:** Tailwind CSS
- **Content:** MDX for case studies and writing
- **Animation:** Framer Motion, used lightly
- **Icons:** Lucide
- **Deployment:** Vercel
- **Forms/contact:** simple email CTA first, form later only if needed
- **i18n:** route-based localization (`/` for English, `/es` for Spanish)

### Why this helps your positioning

It lets the portfolio demonstrate:

- component-based frontend thinking
- structured content architecture
- modern React/Next workflow
- responsive implementation quality
- bilingual product thinking
- maintainable scaling for future case studies

### Recommended product shape

The site should feel like a polished product, not a static brochure:

- strong homepage
- filtered project/case-study pages
- bilingual toggle in header
- polished content blocks
- subtle motion
- professional but modern typography
- diagrams/screenshots when available

### What to avoid even with this stack

- overusing animation
- generic startup gradients everywhere
- dark-mode-only design
- bloated CMS setup too early
- making the portfolio more complex than the projects inside it
